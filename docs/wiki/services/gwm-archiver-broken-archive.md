# winetitles.com.au: broken pre-Jul-2017 GWM article archive

**Date discovered:** 2026-05-23
**Status:** reported to publisher — pending response
**Affects:** 150 issues, Jan 2005 (#492) – Jun 2017 (#641)

## Symptoms

Visiting any pre-Jul-2017 issue page on `winetitles.com.au/gwm/articles/<slug>/`
shows the article list as expected, and each article page renders fine with
an "Article Details" panel containing a working **Download PDF of the article**
form. The form POSTs to `/wp-content/uploads/tmp.php` with `docid` + `dockey`
and the server responds:

```
HTTP/2 200
content-type: application/pdf
content-disposition: attachment; filename="<basename>.pdf"
<0 bytes of body>
```

No error, no redirect, no challenge — just a valid-looking response with an
empty body. Verified across multiple articles in multiple issues; sampled
from a real subscriber session driven via Chrome DevTools MCP so the
session is indistinguishable from a regular user click.

## Boundary

| Issue | Date | Per-article PDF | FULL ISSUE PDF |
|-------|------|-----------------|----------------|
| #641 | Jun 2017 | 0 bytes | absent |
| **#642** | **Jul 2017** | **works (1-3 MB)** | absent |
| #650 | Mar 2018 | works | absent |
| **#651** | **Apr 2018** | works | works (28 MB+) |
| #748 | May 2026 (current) | works | works |

Two transitions visible: Jul 2017 = per-article archive starts working,
Apr 2018 = FULL ISSUE PDF first appears.

## Most likely cause

A WP upload migration somewhere between Jun 2017 and Jul 2017 either dropped
the old `wp-content/uploads/` paths or rewrote URLs without copying the
underlying files. The article-detail form still exists because it's
generated by the post template, so the UI looks unchanged — but the backend
file is gone and `tmp.php` streams an empty response.

## What the publisher would need to fix

Likely options on their side:

1. Restore the `wp-content/uploads/` directory from a pre-2017-migration
   backup, mapping the historical `docid`s to the original PDFs.
2. Re-import each pre-Jul-2017 issue's PDFs into the WP media library and
   re-link them. They almost certainly still have the print PDFs.
3. As a stopgap, set `tmp.php` to return a proper 404 on missing files so
   users (and our archiver) get a clear "not available" signal rather than
   a 0-byte successful response.

## Our archiver's behaviour

`scripts/gwm-archiver.py` does **not** maintain a "known-broken" cache. It
re-probes every dead-range issue every week with fail-fast (2 empties → bail).
Cost is ~9 s per dead issue (1 issue page + 2 article pages + 2 tmp.php
POSTs + sleeps), ~21 min total per weekly run. If the publisher restores
the files, we auto-heal within at most a week — no operator action needed.

## When to retest manually

If you hear back from the publisher that they've restored the archive:

```bash
ssh doc2 sudo systemctl start gwm-archiver
journalctl -fu gwm-archiver
```

A successful recovery would show `synthesised` or `downloaded` status lines
on issues currently flagged `no-pdf`. The OnSuccess Gotify will fire as
soon as the first old issue comes back online.
