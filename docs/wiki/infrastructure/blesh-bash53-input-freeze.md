# ble.sh freezes keyboard input under bash 5.3

- **Date researched:** 2026-06-27
- **Status:** BROKEN upstream (ble.sh dev snapshot) — worked around by disabling ble.sh
- **Affects:** every fleet host (the base profile enabled `programs.bash.blesh`)

## Symptom

You type `bash` from zsh (or otherwise start an interactive bash), the prompt
renders, and **no keystroke registers** — you cannot type anything. zsh is fine
because ble.sh is bash-only.

## Root cause

`/etc/bashrc` (generated by `programs.bash.blesh.enable = true`) sources ble.sh.
The ble.sh version nixpkgs currently ships — `blesh-0.4.0-devel3-unstable-2026-05-28`
— attaches to the terminal but swallows all input under **bash 5.3.9**. Bash
itself is fine; the line editor is the problem.

## Evidence (reproduced in tmux, a faithful terminal emulator)

| ble.sh version | bash 5.3.9 input |
|---|---|
| `0.4.0-devel3-unstable-2026-05-28` (shipped) | **frozen** ❌ |
| `0.4.0-devel3-unstable-2026-03-10` | works ✅ |
| `0.4.0-devel3` (plain) | frozen ❌ |
| ble.sh disabled (`NOSYSBASHRC=1 bash`) | works ✅ |

So it's a **regression in the ble.sh dev snapshot between 2026-03-10 and
2026-05-28**, not a bash bug and not our config.

## Fix applied

`modules/nixos/profiles/base.nix`: `programs.bash.blesh.enable = false`.
Interactive bash now uses plain readline; starship/fzf/atuin/aliases from
`~/.bashrc` still load (they were gated behind readline `vi`/`emacs` mode, which
is on without ble.sh).

## Immediate workaround (no rebuild)

```sh
NOSYSBASHRC=1 bash   # skips /etc/bashrc (and thus ble.sh); ~/.bashrc still loads
```

## When to revisit

Re-enable `programs.bash.blesh.enable` once nixpkgs ships a ble.sh snapshot newer
than 2026-05-28 that fixes the bash 5.3 input freeze (or pin to the working
2026-03-10 snapshot via an overlay if ble.sh in bash is wanted before then).
Verify with the tmux repro above before flipping it back on.
