# Forgejo v0 — minimal self-host on doc2

## Overview

Get Forgejo running on doc2 at `https://git.ablz.au` with a single locked-down account, just enough wiring to be useful (proxy, health monitor, nightly dump to NFS). One real, useful repo lives in it for a week of normal work — issues, PRs, branches, clones, pushes — and that's the bar for "v0 worked."

Everything beyond that — GitHub mirroring, the flake cutover, agent migration, Forgejo Actions, the `tea` CLI, multi-user, federation — is explicitly out of scope and stays deferred until v0 has earned trust.

## Problem Frame

Issue #223 already settled the forge choice (Forgejo over Gitea/GitLab CE). The original plan tried to design the module *and* the cutover *and* the CI runner *and* the agent migration in one go, which over-scoped v0 by a wide margin. This brainstorm walked the module decisions top-down and converged on a much smaller surface that:

- runs Forgejo as production infrastructure for one repo from day one,
- doesn't touch nixosconfig's flake URL or any other host's autoUpgrade,
- avoids ceremony (sops, runner, mirror, declarative admin bootstrap) until v0 has soaked.

## Users

One: me. `DISABLE_REGISTRATION = true` + `REQUIRE_SIGNIN_VIEW = true`. Anonymous visitors see a login wall.

## Requirements

- **R1.** Forgejo reachable at `https://git.ablz.au` with a valid ACME cert and an `A` record managed by `homelab.localProxy` (Cloudflare).
- **R2.** Module at `modules/nixos/services/forgejo.nix` in the `homelab.services.<name>` style; `homelab.services.forgejo.enable = true;` set in `hosts/doc2/configuration.nix`.
- **R3.** State persists at `/mnt/virtio/forgejo` (virtiofs from prom ZFS), owned by a static `forgejo` system user; `DynamicUser = lib.mkForce false`.
- **R4.** SQLite database (no Postgres in v0).
- **R5.** Forgejo's four app secrets (`SECRET_KEY`, `INTERNAL_TOKEN`, `JWT_SECRET`, `LFS_JWT_SECRET`) auto-generated by the upstream module on first run; persist under `${dataDir}/data/.secrets/`. No sops in v0.
- **R6.** First-run admin user created via Forgejo's web installer at first page load. No declarative bootstrap script.
- **R7.** Git transport: HTTPS only. SSH disabled (`server.DISABLE_SSH = true`).
- **R8.** Locked-down policy: no public registration, sign-in required to view, default repo visibility private.
- **R9.** Nightly engine-agnostic dump via `services.forgejo.dump` to `/mnt/data/Life/Andy/Code/forgejo-dumps/`. The `forgejo-dump.service` waits on `mnt-data.mount` (`after` + `requires`) so a stale NFS handle doesn't crash the dump.
- **R10.** `homelab.monitoring.monitors` entry pointing at `https://git.ablz.au/api/healthz` (Forgejo native endpoint). Uptime Kuma alerts via the existing fleet path.
- **R11.** `OFFLINE_MODE = true` (no external avatar/gravatar fetches), `TIME_ZONE = Australia/Perth` to match the rest of the fleet.

## Success Criteria

After deploy:
- `https://git.ablz.au` loads with a valid cert, no warnings.
- `curl -fsS https://git.ablz.au/api/healthz` returns 200.
- Uptime Kuma "Forgejo" monitor green.
- Web installer completes; admin user `abl030` exists.
- Daily dump tarball appears in `/mnt/data/Life/Andy/Code/forgejo-dumps/` and survives the next Kopia sweep.

The week of soak (this is the real test):
- One real, useful repo (a "real, useful" repo per the brainstorm — content TBD on the day of) gets created and pushed to from a workstation over HTTPS.
- Daily activity for ~7 days: issues opened, branches pushed, PRs merged, clones made.
- No service restarts or rebuilds for the week beyond the normal nightly autoUpgrade.

If all of that holds, v0 has earned the right to a v1 (mirror, runner, cutover, etc.) — handled in a separate brainstorm.

## Scope Boundaries

In v0:
- Module + first deploy + one real repo + a week of soak + the dump pipeline.

### Deferred for later (revisit after v0 soak)

- GitHub mirror (either direction). The original plan's pull-mirror→push-mirror inversion is deferred entirely.
- Forgejo Actions runner. No CI in v0.
- Flake-URL cutover (`system.autoUpgrade.flake`). nixosconfig stays on GitHub.
- Agent migration: `.claude/agents/` stays in nixosconfig as today; private agent repos *can* live on git.ablz.au but no code/docs change forces them there yet.
- `tea` CLI; CLAUDE.md updates pointing future agents at Forgejo.
- Sops-managed Forgejo secrets (the four app secrets + admin password).
- Declarative admin bootstrap.
- Postgres via `mk-pg-container` (hostNum 8 stays reserved).
- LFS, package registry, Pages.
- Mailer (SMTP).
- Forgejo Actions with the GitHub-Actions YAML compatibility shim.
- Multi-user (Meg, anyone else); public-read repos.

### Outside this product's identity

- Forgejo as a GitHub replacement for the public-facing world. v0 is a private homelab forge. The "is the public mirror still on GitHub" question is a v1+ topic.
- Federation (ForgeFed). Track upstream maturity; not a v0 concern.

## Key Technical Decisions

- **Storage on virtiofs (`/mnt/virtio/forgejo`)**: matches the doc2 convention for stateful services (paperless, immich) and keeps Forgejo movable to another host with one rsync if doc2's role ever changes.
- **Static `forgejo` user** with `DynamicUser = mkForce false`: forced by the virtiofs decision (DynamicUser conflicts with virtiofs ownership). Pattern mirrors `services/uptime-kuma.nix`.
- **No sops in v0**: the four Forgejo app secrets are per-instance, generated once, never rotated, never shared. Letting the upstream module auto-generate them under `${dataDir}/data/.secrets/` is correct — sops adds ceremony for zero benefit at this scope. Admin password set once via the web installer.
- **HTTPS-only git**: zero firewall change, zero SSH key management. Any client uses `git clone https://git.ablz.au/abl030/<repo>.git` with a Forgejo-issued PAT or password.
- **Port 3023** for HTTP loopback (3000 is convention but 3030 collides with doc2's Grafana). Behind nginx — clients never type it.
- **Backup to NFS, not virtiofs**: `services.forgejo.dump` writes to `/mnt/data/Life/Andy/Code/forgejo-dumps/` so dumps land in the existing Kopia coverage area instead of needing a separate sweep job. Costs us the `after`/`requires` wiring on `mnt-data.mount`. Verify NFS export-side ownership lets the `forgejo` user write at deploy time.
- **`pkgs.forgejo-lts` package**: upstream default on stable channels, longer upgrade runway, fine for v0.
- **Module options surface = `enable` + `dataDir`**: matches the paperless/uptime-kuma convention. Add more options when a second host wants to vary.

## Open Questions

### Resolved during brainstorming

- DB? → SQLite.
- Storage? → virtiofs `/mnt/virtio/forgejo`.
- User identity? → static `forgejo` (forced by virtiofs).
- Secrets? → none in v0 (auto-gen + web installer).
- Git transport? → HTTPS only.
- FQDN? → `git.ablz.au`.
- HTTP port? → 3023.
- Backup? → nightly dump → `/mnt/data/Life/Andy/Code/forgejo-dumps/`, swept by existing Kopia.
- User policy? → just me, locked down.
- Mailer? → none.
- `OFFLINE_MODE`? → true.

### Deferred to deploy time

- The exact identity of the test repo (greenfield vs migrated subset of `.claude/agents/` vs other). Decide on the day of; doesn't change the module.
- Whether NFS export-side permissions on `/mnt/data/Life/Andy/Code/` already let the `forgejo` system user write. Verify post-deploy; fix the export if not.

### Deferred indefinitely (until proven needed)

- Whether v0 ever needs to grow into v1 (mirror, runner, cutover) at all. Possibly the right outcome is "v0 is enough — agent repos live here, nixosconfig stays on GitHub forever, no cutover is ever justified."

## Module Sketch (the v0 surface, for planning reference)

The full module shape is in the brainstorm conversation — ~55 lines, single file, no helpers. Key structure:

- `options.homelab.services.forgejo = { enable; dataDir = "/mnt/virtio/forgejo"; };`
- `services.forgejo` with `package = forgejo-lts`, SQLite, dump enabled, locked-down `service`/`repository` settings, `OFFLINE_MODE = true`, `TIME_ZONE = "Australia/Perth"`.
- Static user/group, `DynamicUser = mkForce false`, `ReadWritePaths = [ dataDir, dumpDir ]`.
- `forgejo-dump.service` gains `after`/`requires` on `mnt-data.mount`.
- `tmpfiles.rules` for `dataDir`, `${dataDir}/data`, and the dump dir.
- `homelab.localProxy.hosts = [{ host = "git.ablz.au"; port = 3023; websocket = true; }]`.
- `homelab.monitoring.monitors` with the `/api/healthz` URL.
